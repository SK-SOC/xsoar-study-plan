XSOAR Architecture — 5 Things You Must Know

The 5 Core Components

COMPONENT 01
🖥️ The Server
The core brain. Hosts the UI, stores all incidents and indicators, runs playbooks, and manages all integrations. Everything lives and runs here.
Always required
COMPONENT 02
⚡ Engines
Remote workers. If your XSOAR server is cloud-based but your SIEM is on-prem behind a firewall, an Engine sits on-prem and acts as a secure relay. No firewall changes needed.
For hybrid environments
COMPONENT 03
🔌 Integrations
API connections to external tools — VirusTotal, Splunk, CrowdStrike, Jira, Slack. Each integration exposes commands you call inside playbooks. Configure once, use everywhere.
API connection wrappers
COMPONENT 04
📋 Playbooks
Your automated runbooks. Visual drag-and-drop logic flows. If an alert comes in → run these steps in this order → make these decisions. This is where you'll spend most of your time.
Your main output
COMPONENT 05
🛒 Marketplace
XSOAR's app store. 700+ pre-built integrations and playbooks — VirusTotal, MISP, PAN-OS, CrowdStrike, Jira, Slack, and hundreds more. Most are free to install in minutes instead of building from scratch.

Lesson 3 of 3 // War Room

The War Room — XSOAR's Most Unique Feature

Every incident has a live, automatic audit trail. Nothing else in the SOC works like this.

Learning Objectives
  • Understand what the War Room is and what gets logged automatically
  • Know how to run commands directly inside the War Room
  • Understand why this matters for analyst workflow and audit trails

What Is the War Room?

Every incident in XSOAR has its own War Room — a live, chronological log of everything that happens during investigation. Think of it as three things combined into one:

💬 Group Chat
Analysts can post comments, observations, and decisions directly in the incident thread
📜 Audit Log
Every automated action is timestamped and recorded — no one needs to write a report
⌨️ Command Terminal
Type commands directly to query tools live, right inside the incident context

Knowledge Check // 5 Questions

Test What You've Learned

No tricks — this is to confirm understanding before the lab. Select one answer per question.

Question 1 of 5
Your analyst spends 40 minutes manually querying VirusTotal, AbuseIPDB, and Shodan for every alert. Which SOAR capability eliminates this specific pain?
✓ Correct! Automation means tasks run without a human doing them manually. Your OSINT enrichment workflow is a perfect automation use case — XSOAR queries all sources simultaneously, in seconds.
✗ Not quite. Response is about taking action (block, isolate, notify). Orchestration is about connecting tools together. What you're describing — queries running without a human doing them — is Automation.
Question 2 of 5
You want XSOAR to query the VirusTotal API when a phishing alert fires. Which XSOAR component wraps the VirusTotal API and exposes it as a command you can use in a playbook?
✓ Correct! An Integration is the API connection wrapper. You configure it once with your API key, and it exposes commands like !vt-get-ip-report that any playbook can call.
✗ Close — a Playbook uses integrations but isn't the connection itself. An Engine is for on-prem relay. The War Room is the investigation log. The component that wraps an API is an Integration.
Question 3 of 5
Your XSOAR server is cloud-hosted. Your SIEM is on-premises behind a strict firewall with no inbound rules. How does XSOAR still query your SIEM?
✓ Correct! An Engine sits on-premises and acts as a relay. The XSOAR cloud server sends instructions to the Engine, which executes them locally and sends results back. No inbound firewall rules needed.
✗ The answer is an Engine — a remote worker installed on-premises that acts as a secure relay. The connection is outbound from the Engine to XSOAR, so no inbound firewall rules are required.
Question 4 of 5
A new analyst asks: "Where do I test commands before putting them into a live playbook?" Which XSOAR feature do you point them to?
✓ Correct! The Playground is a sandbox where analysts can run commands and test scripts safely. Nothing in the Playground affects real incidents. It's the equivalent of a staging environment for commands.
✗ The Playground is the sandbox for safe command testing. The Incidents view is for real alerts. Marketplace is for installing content. Settings is for configuration. The Playground is your safe test environment.
Question 5 of 5
After a security incident is closed, your CISO asks for a full audit trail of everything that happened during the investigation. In XSOAR, where does this automatically live?
✓ Correct! The War Room automatically logs every action — playbook steps, integration queries, analyst comments, decisions — all timestamped. Your CISO gets a perfect audit trail with zero extra effort from the analyst.
✗ The answer is the War Room. Every action — automated and manual — is timestamped and logged there automatically. No separate reporting tool needed. It's one of XSOAR's most valuable features for audit and compliance.

Day 1 Glossary — Key Terms

SOAR
Security Orchestration, Automation and Response — platform category that XSOAR belongs to
Integration
XSOAR's term for an API connection to an external tool (VT, SIEM, EDR, Slack, etc.)
Playbook
Automated runbook — visual logic flow that runs tasks when an incident is triggered
War Room
Per-incident live log — auto-records every action, command, and analyst note with timestamps
Engine
On-prem relay worker that lets cloud-hosted XSOAR reach tools behind a firewall
Playground
Sandbox environment for testing commands safely — nothing here affects live incidents
Marketplace
XSOAR's app store — 700+ pre-built integrations and playbooks, most free to install
Incident
XSOAR's core work unit — an alert becomes an incident, gets enriched, worked, and closed
