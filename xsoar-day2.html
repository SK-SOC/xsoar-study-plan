What Is an Incident in XSOAR?

Every alert that enters XSOAR — from your SIEM, EDR, email gateway, or manually — becomes an Incident. It is the core work unit that everything else revolves around.

Think of it as a structured case file — it holds all the data, all the enrichment results, all the analyst notes, and drives the automated playbook from start to finish.

Lesson 2 of 3 // Indicators & IOCs

Indicators — XSOAR's IOC Intelligence Store

Indicators are the threat intelligence objects that link your incidents together. Every IP, URL, hash and domain lives here.

Learning Objectives
  • Understand what XSOAR Indicators are and how they differ from incident fields
  • Know the DBot Score system — the reputation scale XSOAR uses for every IOC
  • Understand how indicators link to incidents automatically

What Is an Indicator?

An Indicator is XSOAR's term for an IOC — any observable artifact that might be associated with a threat. Every time a playbook enriches an IP, URL, file hash, or email address, XSOAR creates an indicator entry and stores it in the Indicators database.

🔑
Key difference: Incident fields hold data about a specific case. Indicators are global objects shared across all incidents. If IP 185.220.101.45 appears in 10 different incidents, there is one Indicator entry that links to all 10.

The DBot Score — XSOAR's Reputation System

Every indicator gets a DBot Score — a reputation rating from 0 to 3 that tells you how malicious it is. This score is set automatically by your integrations (VirusTotal, AbuseIPDB, etc.) and drives playbook decision-making.

0
Unknown
Never seen before — no reputation data available. Treat with caution.
1
Good
Confirmed safe. Example: 8.8.8.8 (Google DNS) returns DBot Score 1.
2
Suspicious
Potentially malicious. Needs more investigation. Escalate for review.
3
Malicious
Confirmed threat. Playbook should auto-block and escalate immediately.

How DBot Score Drives Playbook Decisions

Your OSINT enrichment playbook will use DBot Score as the branching condition. This is the exact logic you will build in Week 3:

Playbook Logic — DBot Score Branching
# Playbook checks DBot Score after enrichment IF DBot Score == 3 (Malicious): → Auto-block IP on firewall → Isolate endpoint on EDR → Escalate severity to Critical → Page on-call analyst ELIF DBot Score == 2 (Suspicious): → Flag for analyst review → Set severity to High → Send Slack notification to SOC channel ELIF DBot Score == 1 (Good): → Mark as False Positive → Auto-close incident → Add to whitelist ELSE DBot Score == 0 (Unknown): → Queue for manual analyst review

Knowledge Check // 5 Questions

Test What You've Learned

Day 2 concepts — incidents, indicators, DBot Score, and Marketplace.

Question 1 of 5
A phishing alert fires from your SIEM. VirusTotal returns a score of 8/10 engines flagging the source IP as malicious. What DBot Score does XSOAR assign to this indicator?
✓ Correct! 8/10 antivirus engines flagging is a confirmed malicious verdict — DBot Score 3. Your playbook should react to this by auto-blocking and escalating to Critical severity.
✗ DBot Score 3 = Malicious. When the majority of engines flag an IOC, that's a confirmed threat. Score 2 (Suspicious) would apply when only a few engines flag it and there's uncertainty.
Question 2 of 5
IP address 185.220.101.45 appears in 12 different phishing incidents in your XSOAR instance. How many Indicator entries exist for this IP?
✓ Correct! There is exactly ONE indicator entry for that IP. It links to all 12 incidents. This is what makes the Indicators database powerful — you can see all incidents involving a threat actor's IP in one place.
✗ One indicator entry exists, linked to all 12 incidents. Indicators are global objects — not duplicated per incident. This lets you pivot across all cases involving the same IOC instantly.
Question 3 of 5
An alert arrives in XSOAR from the SIEM. No analyst has touched it yet. The playbook is about to start running. What is the correct incident status at this moment?
✓ Correct! Active means the playbook is running and auto-enrichment is in progress. New is the very first state before any processing starts. In Progress is when an analyst manually picks it up.
✗ Active is correct here. The lifecycle goes: New (just arrived) → Active (playbook running) → In Progress (analyst working it) → Pending (waiting for input) → Closed. The playbook running means it's Active.
Question 4 of 5
You install the VirusTotal Content Pack from Marketplace. You then go to the War Room and run !vt-get-ip-report ip="8.8.8.8" — but get an error saying "integration not configured." What did you miss?
✓ Correct! Installing from Marketplace only downloads the integration code. You must then go to Settings → Integrations, find VirusTotal, click "Add instance," and enter your API key. Then test it — only after a passing test does the command work.
✗ The missing step is adding an integration instance. Marketplace installs the code, but you still need to configure credentials. Go to Settings → Integrations → find VirusTotal → Add instance → enter API key → Test → Save.
Question 5 of 5
You want XSOAR to automatically check AbuseIPDB AND VirusTotal AND Shodan for every new phishing incident — and store all results in one place for the analyst. Which combination of XSOAR features makes this happen?
✓ Correct! This is exactly the OSINT enrichment playbook you will build in Week 3. Install 3 integrations, build one playbook with 3 parallel tasks calling each command, and all results flow into the incident context and War Room automatically.
✗ The answer is option D — integrations provide the API connections, and a playbook orchestrates all three calls together. Results land in incident context (structured data) and War Room (readable log). This is your Week 3 capstone lab.

Day 2 Glossary

Incident Type
Category that determines which playbook auto-runs and which fields are shown (Phishing, Malware, Network, etc.)
Severity
6-level scale (Unknown → Critical) that drives analyst priority and can be auto-updated by playbooks
Indicator
Global IOC object (IP, URL, hash, domain) — shared across incidents, holds DBot Score and enrichment data
DBot Score
XSOAR reputation scale: 0=Unknown, 1=Good, 2=Suspicious, 3=Malicious — drives playbook branching
Content Pack
Marketplace bundle containing an integration + playbooks + scripts + incident types for one vendor
Integration Instance
A configured copy of an integration with specific API credentials — one integration can have multiple instances
Incident Lifecycle
New → Active → In Progress → Pending → Closed — tracks where in the workflow each case sits
IOC
Indicator of Compromise — any artifact (IP, hash, URL, domain) associated with a threat actor or campaign
